# Changelog

All notable changes to `ospex-market-maker` are recorded here. The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and versions follow [semver](https://semver.org/) — pre-1.0, a minor bump may change the CLI or config surface.

## [Unreleased]

### Added

- **Single-process `state.dir` lock.** `run` and `cancel-stale` now acquire an `O_EXCL` lock file (`state.dir/maker.lock`) at boot, so a second MM against the same `state.dir` is **refused** (the refusal names the live holder — pid, host, maker wallet, run id) instead of silently last-writer-wins-corrupting `maker-state.json` (the corrupted blob would still pass the shallow load validation, so the state-loss fail-safe never fired). A stale lock left by a crashed run is reclaimed automatically only when its recorded process is verifiably dead on the same host; a cross-host or unparseable lock fails closed (remove `maker.lock` by hand after confirming no MM is running). Read-only commands (`status`, `doctor`) take no lock. Enforces the long-documented "one MM per state directory" rule (DESIGN §12) and is the mechanism behind cancel-stale's "stop `run --live` first" guidance. See `OPERATOR_SAFETY.md`.
- `ospex-mm candidates [--sport <sport>] [--hours <n>] [--json]` — a **read-only, signer-free** operator preflight (no keystore, no passphrase prompt, no writes) that lists **both quote candidates and setup candidates** in one pass. **Market-aware:** a *verified* contest is classified **per configured market** (`marketSelection.markets`, default `['moneyline']`) — one item per `(contest, market)`. Classifications: `quote_ready` (verified contest + open speculation for that market + reference odds when `requireReferenceOdds`), `needs_speculation` (verified, no open speculation for that market), `needs_verification` (contest created but not yet verified), `setup` (game upcoming with `contestCreated=false`, `canCreateContest=true`, `hasOdds=true`), or `skipped` with a reason (`started-or-live` / `no-odds` / `no-reference-odds` / `cannot-create-contest` / `deny-list` / `not-quotable-status` / `game-status-postponed-or-cancelled`). Per-market kinds carry `market` + `speculationId`; contest-level kinds carry `market: null`. `--json` emits a `{ schemaVersion: 2, candidates: … }` envelope (items sorted by kind priority, then matchTime, then market; `truncated: true` when a pagination bound was hit). `--hours` accepts 1–720 (the games API max); the contests leg is capped at the contests API max of 168h — `config.contestsHours` in the envelope carries the effective value — so a long window still returns the full games/setup side instead of failing on the narrower contests endpoint. The allow-list annotates contest-backed items with `inContestAllowList` but never hides them; the deny-list skips. An empty listing is a valid answer and exits 0. See `AGENTS.md` §2.7 for the envelope schema.
- **Spread + total markets (opt-in).** `marketSelection.markets` now accepts `spread` and `total` alongside `moneyline`; the default stays `["moneyline"]`. When the tracking cap is tight the list order is the priority (earlier-listed markets win slots). Each tracked market opens its own reference-odds stream. See `OPERATOR_SAFETY.md` "Markets — spread / total".
- **`status` per-market breakdown.** The `status` report now carries a per-market view of both commitments and positions: `commitments.byMarket` (count per `moneyline` / `spread` / `total`), `positions.byMarket` (count + own-risk USDC per market), and `commitments.distinctSpeculationsNonTerminal` — the count of distinct markets + lines with matchable exposure (a `speculationId` is 1:1 with a contest's market + line on chain), distinct from the existing `distinctContestsNonTerminal` (a contest may now carry several markets). The text renderer adds a `by market:` line under Commitments and Positions, and the Commitments headline now shows both the distinct-contest and distinct-market counts. Additive — the `--json` envelope stays `schemaVersion: 1` and a moneyline-only maker sees the same totals (all exposure under `moneyline`).
- **`summary` per-market fill rate + realized P&L.** Live telemetry events now carry a `market` tag (`'spread'` / `'total'`; **omitted for moneyline**, so a moneyline-only run's NDJSON is byte-identical) on `submit` / `replace` / `soft-cancel` / `fill` / `settle` / `claim` (and the `would-*` forms) — the same convention as the per-market `candidate` event. `summary` reads it to add `liveMetrics.fills.byMarket` (quoted / filled / fill-rate per market) and `liveMetrics.realizedPnl.byMarket` (net + outcome counts per market); each is zero-filled and the run-wide totals equal the sum across markets, so an older moneyline-only log buckets entirely under `moneyline`. The text report prints a `by market:` line under Fills and Realized P&L on a multi-market run (suppressed when only one market is active). Additive — the `--json` envelope stays `schemaVersion: 1` and the NDJSON line shape is unchanged (the new `market?` tag is omitted for moneyline, so moneyline-only logs are byte-identical).

### Changed

- **Retired the dead `marketSelection.requireOpenSpeculation` flag; added `marketSelection.seedSpeculations` (default `false`).** `requireOpenSpeculation` was parsed but never read — the MM's "only quote existing open speculations" behavior was hard-wired. Its replacement, `seedSpeculations`, is the explicit opt-in to **seeding**: posting at the oracle-primary line for a market with no open speculation yet, lazily creating it on first match (which makes the MM actively create speculations, pay the protocol creation fee, and approve the `TreasuryModule`). The flag is staged ahead of its behavior — the seeding path ships in subsequent changes — so for now enabling it is **refused at boot** with a clear "not yet implemented" message rather than silently no-op'd. The retired key is rejected with a loud unknown-field error (no silent fallback) — remove it from your config.
- **Renamed `marketSelection.maxTrackedContests` → `maxTrackedMarkets`.** The cap counts tracked `(contest, market, line)` entries, so with more than one configured market a single contest can consume several slots. The former key is now rejected with a clear unknown-field error (no silent fallback) — rename it in your config.
- `OspexAdapter.listGames` over the SDK's `client.games.list` (`@ospex/sdk` ≥ 0.6.x games surface) — full-schedule reads pass `availableOnly: false` explicitly; the games rows' provider-named `externalIds` are dropped at the adapter boundary (`GameView`) and never surfaced in CLI output.
- **One-wallet-per-instance diagnostics.** The `state.dir` lock enforces one process per state directory but cannot enforce one *wallet* across directories; two instances on the same keystore cause mutual quote-cancellation and `unknown-own-fill` cold-restart churn. Two cheap detections now surface that misconfiguration: (1) once the live signer is unlocked, the resolved wallet is **stamped into the lock file** (`updateMaker`) so the lock's diagnostics + fail-closed refusal name the real wallet — previously commonly `null` for a Foundry keystore (whose plaintext address is omitted); and (2) a **live boot reads the wallet's open commitments** and, if any were not posted by this instance, emits a `foreign-maker-commitments` telemetry event and a loud WARN (best-effort, WARN-only — a foreign open commitment can also be benign prior-run residue or a hand-posted quote; never refuses). New `OPERATOR_SAFETY.md` "One wallet per instance" section + DESIGN §12 note.

### Changed

- **SSE stream-budget guardrail is now per-host-aware AND owner-reserve-aware.** A single instance opens `odds.maxRealtimeChannels` **anonymous** odds streams **+ exactly one** composite **owner-auth** own-state stream (the own-state stream count is corrected from 3 to 1 — it ships as one composite connection, not three). The core-api per-IP cap (default 16) is **split**: it reserves `RESERVED_STREAM_CONNECTIONS_PER_IP_OWNER` slots (default 3) for owner-auth, so anonymous odds streams may use only `cap − reserve` (default 13) — the **binding** budget, tighter than the total cap. The boot guardrail now warns on **both** constraints independently: the anonymous-odds budget (`maxRealtimeChannels > cap − reserve` — e.g. `maxRealtimeChannels = 14` now warns and would 429 on core-api, which the prior total-only check missed) and the total cap (`maxRealtimeChannels + 1 > cap`); plus it always logs a per-host reminder (the cap is per **egress IP/host**, shared across co-located instances this process can't see). The mirrored default cap is raised 10 → 16 to match core-api. New `docs/DESIGN.md` "SSE connection budget" section documents the anonymous/owner split and both per-host formulas (`N·maxRealtimeChannels ≤ cap − reserve` for odds, `N·(maxRealtimeChannels+1) ≤ cap` overall), the `odds.subscribe: false` polling-fallback lever, and the deferred odds-stream-multiplexing plan.
- **Doc-truth pass + a clearer `summary` latent-exposure-peak line.** Corrected stale documentation to match the shipped (live-capable, own-state-SSE) architecture: CONTRIBUTING's "no live write paths" rule (live `submit` / cancel / approve ARE implemented, gated behind the two-key live model — not absent); the `summary` and `run` module docstrings (which described the dry-run-only / per-tick-polling world as current — own-state is now delivered over the owner-auth SSE stream, canonical since OS-Phase 4, with the per-tick probes demoted to a 60s audit cross-check); the README test count; and a removed `OSPEX_MM_*` env-override claim in `docs/DESIGN.md` (never implemented). The `summary` CLI's **Latent-exposure peak** line now states it is derived from dry-run `would-*` events and reads 0 on a live run, so a live `0.000000` is no longer misread as "no latent exposure occurred."

### Fixed

- **Discovery now pulls a departing contest's visible quotes off the book before untracking it.** When a tracked contest drops out of the listing window (started / scored / out of window), the discovery cycle soft-cancels its `visibleOpen` quotes (and retains / on-chain-cancels any partially-filled remainder) instead of stranding them on the relay book — unpriced and still matchable — until expiry. Previously a contest could leave the listing without first passing through an unquoteable gate, so nothing took its quotes down; harmless under short expiries, but dangerous under `match-time` expiry where the window can be hours. Upholds the "the visible book must never carry a quote the MM is no longer pricing" invariant (DESIGN §2.2 / §3) on the one teardown path that skipped it. If that untrack-time cancel fails transiently, the market is **retained** (flagged `departing`) and the pull is retried every tick — pulled-only, never re-quoted — until it drains, then untracked on a later discovery cycle (so a failed cancel can't strand a still-matchable quote with nothing tracking it).
- **Tracked markets' `matchTimeSec` / `speculationId` are refreshed from the latest listing on every discovery cycle.** They were frozen at first-tracking, so a rescheduled start — especially one moved **earlier** — was not reliably caught by the `start-too-soon` / `match-time`-expiry gates (the odds stream keeps the freshness timer fresh, so the stale-reference gate did not bound it). The refresh reuses the listing already in hand, so it adds no extra reads.
- **Own-state: a truncated-snapshot paging failure no longer leaves phantom rows in the canonical book.** When a cold-connect's snapshot is still paging (its latest page was `truncated: true`) and the transport reconnects (`onStatus('reconnecting')`), the partial accumulation is now discarded so the next cold snapshot **replaces** it instead of being merged onto it. Previously the pinned SDK (v0.6.2) recovered a mid-flight REST paging failure by clearing its cursor and cold-restarting while emitting `reconnecting` (not `resync`), and the runner merged the fresh snapshot's first page onto the stale partial — so rows already filled/cancelled by snapshot time could swap in as phantom canonical quotes. The discard is gated **precisely on a `truncated` in-flight baseline** — the SDK never advances its running cursor on a truncated page, so that case always cold-restarts. A **complete** (`truncated: false`) baseline is **kept**: the SDK advanced its cursor, so a drop-before-`ready` reconnects as a resume whose following `onReady` swaps the held baseline (discarding it there would mark the session ready over empty state and under-count owned exposure). Never triggered on `degraded` (overloaded with `positionsTruncated`). The complete fix is SDK-side (emit `resync` on paging failure); this is the consumer-side stopgap until that lands.
- **A shared `telemetry.logDir` no longer false-trips the boot state-loss hold across instances.** Every telemetry line on a live run — from **both** `run --live` and the one-shot `cancel-stale` — is now stamped with the instance's `maker` wallet (a public on-chain address — additive `maker` field, also a new reserved key), and both commands' boot-time state-loss fail-safe scopes its "a prior run happened" check to *this* maker (`cancel-stale` unlocks the signer to resolve the maker before that scoped check — a Foundry keystore reveals the address only after the unlock). Previously the check counted **any** `run-*.ndjson` in the directory, so a second instance booting fresh into a shared log dir was forced into the state-loss hold by a *sibling's* logs — and the only escape was `--ignore-missing-state`, the exact override that defeats the real fail-safe. A genuine state loss for the same wallet still holds (verified by test); a sibling's logs (different maker), and unattributable maker-less legacy logs, do not. Dry-run (no signer) keeps the original directory-wide check (it carries no on-chain exposure). One-time migration note: a live run upgraded from a pre-this-release version won't count its own pre-existing maker-less logs — rely on the `state.dir` lock / state file across that boundary. Docs: new `OPERATOR_SAFETY.md` "Running more than one MM" section covers per-instance `state.dir` / `telemetry.logDir` / `killSwitchFile`, the **KILL-file semantics at N>1** (a shared `./KILL` is a fleet kill; a per-instance file has no fleet-wide switch; latency is up to one tick — SIGINT/SIGTERM is the fast per-process stop), and the per-host SSE budget.

- **A book-hidden commitment's captured signed payload is no longer dropped by a later own-state delta that omits it.** `reduceOwnerCommitmentObservation` wholesale-replaces the canonical commitment record on every SSE `commitment` delta, previously carrying forward only the `fills[]` array. So a delta arriving with `signedPayload: null` (e.g. an indexer-sourced `filledRiskWei6` bump that doesn't re-include the signature) downgraded a previously-captured record to `signedPayloadStatus: 'missing-legacy'` and discarded the bundle — the only handle for an authoritative on-chain cancel of an off-book (book-hidden) row, leaving it strandable with no `cancelOnchain` path. The replace now also preserves a previously-captured `signedPayload` (and its `'present'` status) when the incoming delta lacks one. The payload is immutable for a given `commitmentHash` (the hash is the EIP-712 digest of the signed struct), so reusing the prior is always correct; the carry-forward is upgrade-only (missing→present, never the reverse), a delta that DOES carry the payload still uses its own, and the state validator's `present` ⟺ payload-set-with-matching-hash invariant is maintained.

## [0.1.0-alpha.1] — 2026-06-12

First tagged release. The full v0 command surface is implemented (793 unit tests) and the live path has been exercised end-to-end on Polygon mainnet in small, controlled runs — posting, partial fills, soft and on-chain cancels, expiry, stream reconnects, process restarts, and the score → settle → claim lifecycle. Distribution is clone-and-run at this tag; nothing is published to npm.

### Added

- The complete CLI surface: `doctor`, `quote --dry-run`, `run --dry-run`, `run --live`, `cancel-stale [--authoritative]`, `status`, and `summary` — each with a `--json` envelope where applicable; see the README "Current status" section and `AGENTS.md` for the report schemas.
- Pricing (vig strip → fair value → spread → two-sided quote), a worst-case-loss-by-outcome risk engine with per-commitment / contest / team / sport / bankroll caps, the order reconcile, persistent single-process JSON state, and NDJSON telemetry with `summary` aggregation including realized P&L.
- Live mode is own-state-stream-driven: the owner-authenticated SSE stream is the canonical writer of the maker's commitments, fills, and positions; per-tick polls survive only as an audit cross-check.
- Operator safety rails: two-key live gating (`mode.dryRun: false` AND `--live`), the boot-time state-loss fail-safe, wallet-bounded boot auto-approve (`exact` mode default), a daily POL gas budget with an emergency reserve, the funding guard, auto-settle / auto-claim, and an off-chain (plus optional on-chain) cancel sweep on shutdown.

### Alpha safety envelope

- **Moneyline only**, and only on speculations that already exist — the MM is not a speculation seeder.
- **Dry-run first.** `mode.dryRun: true` is the default; go live deliberately with tiny caps.
- **Single-process local state** — one MM process per state dir / maker wallet; the JSON state file is not multi-process safe.
- **An off-chain cancel removes a quote from the public book but does NOT invalidate the signed payload** — a taker already holding it can still match it until expiry (default ≈ 2 minutes). Remainder safety after a partial fill therefore relies on short expiries or the on-chain paths (`killCancelOnChain: true`, `cancel-stale --authoritative`).
- **No unrealized P&L yet** (realized P&L is wired); no `raiseMinNonce` bulk invalidation — on-chain cancels are per-commitment.
- **Bring your own RPC URL and Foundry keystore** — there are no defaults, and raw private keys are never accepted. `wallet.keystorePath` is used verbatim (`~` is not expanded; use an absolute path) and can stay blank until you go live.

### Pinned

- `@ospex/sdk` v0.6.2, resolved from its GitHub Release tarball (`package.json` / `yarn.lock` are the source of truth). Node ≥ 20.
