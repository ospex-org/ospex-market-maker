# ospex-market-maker

A **reference market maker** for [Ospex](https://ospex.org) — a zero-vig peer-to-peer sports prediction protocol on Polygon. Clone it, point it at your wallet, bankroll, RPC URL, and return target, and it quotes two-sided liquidity on upcoming contests, manages its exposure, reacts to fills, settles and claims, and writes an auditable record of everything it did.

It is built on **[`@ospex/sdk`](https://github.com/ospex-org/ospex-sdk)** — every chain and API interaction goes through the SDK; this repo never calls the Ospex contracts directly.

---

> ## ⚠️ Status — alpha
>
> The full command surface and live execution path are implemented, and the live path has been exercised end-to-end on Polygon mainnet in small, controlled runs — visible and partial fills, soft and on-chain cancels, expiry, stream reconnects, process restarts, and the score → settle → claim lifecycle. It remains **experimental software**: run it in dry-run first, go live deliberately with tiny caps, and read **[`docs/OPERATOR_SAFETY.md`](docs/OPERATOR_SAFETY.md)** before risking funds. The full design is in **[`docs/DESIGN.md`](docs/DESIGN.md)** — the command surface, config schema, pricing model, risk model, and lifecycle are specified there.

## Current status

What works:

- `yarn install && yarn build && yarn typecheck && yarn lint && yarn test` — clean (888 unit tests across the pricing, config, risk, orders, state, telemetry, ospex-adapter, reducer, runner, and CLI-command modules).
- **`ospex-mm doctor`** — readiness probe: config, keystore, API, RPC, POL/USDC balances, `PositionModule` allowance, and the persisted-state integrity check, plus a "Ready to" matrix (dry-run shadow — needs no keystore, since the shadow loop posts nothing / post commitments — the live prereqs). `--address <0x…>` keeps it fully read-only (no passphrase prompt); `--json` emits a `{ schemaVersion: 1, doctor: … }` envelope.
- **`ospex-mm quote --dry-run <contestId>`** — computes a two-sided moneyline quote breakdown (reference odds → fair value → spread → priced quote with size), or refuses with a clear message (contest closed, no open moneyline speculation, no reference odds). Never posts; `--json` emits a `{ schemaVersion: 1, quote: … }` envelope.
- **`ospex-mm run --dry-run`** — the shadow loop: boots through the state-loss fail-safe, then runs the tick loop — discovery → reference-odds tracking → the per-market reconcile → age-out → state flush — until a kill-switch file appears or a SIGTERM / SIGINT arrives; logs `quote-intent` / `quote-competitiveness` / `would-submit` / `would-replace` / `would-soft-cancel` / `candidate` / `expire` / `degraded` / `error` / `kill` events to the NDJSON log under `telemetry.logDir`, and mutates a *hypothetical* inventory — but **posts nothing on chain**. `--address` / `--keystore` (overrides the configured path) / `--ignore-missing-state` (attests no prior run left a still-matchable commitment — lifts the boot-time hold).
- **`ospex-mm cancel-stale [--authoritative]`** — one-shot operator cleanup: pulls every still-matchable tracked commitment whose age exceeds `orders.staleAfterSeconds` (records already past expiry are skipped — dead on chain). By default it runs the gasless off-chain DELETE for each stale `visibleOpen` (records → `softCancelled`, emits `soft-cancel` `reason: 'stale'`); a `partiallyFilled` remainder is **skipped** off-chain — the API rejects a DELETE once a commitment has matched (`409 COMMITMENT_MATCHED`), so it's counted in `offchainSkippedPartial` and left in place (its `filledRiskWei6` untouched) — and `softCancelled` records are a no-op off-chain, so for either only `--authoritative` invalidates the still-matchable signed payload. With `--authoritative` it *also* calls `cancelCommitmentOnchain` per non-terminal record — including `partiallyFilled` (records → `authoritativelyInvalidated`, emits `onchain-cancel`) — gas-gated with `mayUseReserve: true` (operator-explicit "burn the reserve to kill latent exposure"), denials emit `candidate` `gas-budget-blocks-onchain-cancel` and break the loop. Stop a running `run --live` first — the JSON state file is not multi-process safe. Same two-key principle as `run --live` (requires `mode.dryRun: false` in the config; the keystore passphrase comes from `OSPEX_KEYSTORE_PASSPHRASE` or a TTY prompt). Mirrors the runner's boot-time state-loss fail-safe (DESIGN §12): refuses if the state file is missing but prior telemetry exists, unless `--ignore-missing-state` is passed, and refuses on a dry-run synthetic hash (`dry:…`) in the loaded state. Both checks run *before* the keystore unlock so a doomed invocation never prompts for a passphrase. `--json` emits a `{ schemaVersion: 1, cancelStale: … }` envelope; exits non-zero if any record failed to cancel (`errored > 0`), a gas-budget denial blocked the on-chain leg (`gasDenied > 0`), or a legacy book-hidden record had no recoverable signed payload so the authoritative cancel was blocked (`blockedMissingPayload > 0`).
- **`ospex-mm status [--address <0x…>] [--json]`** — read-only snapshot of the persisted state: commitments by lifecycle (`visibleOpen` / `softCancelled` / `partiallyFilled` / `filled` / `expired` / `authoritativelyInvalidated` counts + distinct contests *and* distinct speculations/markets with non-terminal exposure), positions by local `MakerPositionStatus` (counts + per-bucket own-risk USDC sums), a per-market breakdown of both commitments and positions (moneyline / spread / total), today + lifetime gas spent (POL) / estimated seed creation-fees (USDC — a conservative estimate, not exact realized spend), the PnL snapshot, last flushed-at + last run id, and — when a maker address resolves without decrypting the keystore (`--address`, or a v3 keystore with a plaintext `address` field) — the SDK's `positions.status(maker)` totals from the API (signer-free read). No keystore unlock; no writes; always exits 0. State-loss situations (missing-but-prior-telemetry, corrupt file) are **reported**, not refused — the integrity field surfaces the diagnostic so the operator can see *why* their MM is misbehaving. `--json` emits a `{ schemaVersion: 1, status: … }` envelope.
- **`ospex-mm run --live`** — the same loop, with the reconcile's submits going through `commitments.submitRaw` and the off-chain cancels through `commitments.cancel` (the protocol tuple comes from `toProtocolQuote`; nonce auto-picked; the short fixed-seconds / match-time `expiry` is passed explicitly). Gated by the two-key model (DESIGN §8): requires *both* `mode.dryRun: false` in the config *and* the `--live` flag; otherwise refused with a clear message. Reads the keystore passphrase from `OSPEX_KEYSTORE_PASSPHRASE` (preferred for non-interactive runs), else prompts no-echo on a TTY. The signer's address determines the maker wallet — `--address` with `--live` is refused. The Runner ctor also refuses to boot live on a state directory containing dry-run synthetic commitments (use a fresh `state.dir` for live). Events log as `submit` / `replace` / `soft-cancel` (instead of the `would-*` counterparts), carrying the real commitment hashes. **Own-state is stream-driven** (DESIGN §10): the owner-authenticated own-state SSE stream — always on in live mode — is the canonical writer of the maker's commitments, fills, and positions, so fills and lifecycle changes land in real time (the affected market is dirtied and re-priced by the next reconcile), `fill` events carry `source: 'own-state-stream'`, and each position's forward status step emits a `position-transition` event (`claimed` is stamped later by the auto-claim path). The per-tick **audit probes** — the `listOpenCommitments(maker)` diff, one aggregate `getPositionStatus(maker)`, and per-hash probes of soft-cancelled records — survive only as a slow cross-check: they converge a per-tick audit clone for the divergence comparator and never write canonical state or gate the tick (a probe failure marks the audit cycle failed; the comparator skips it and preserves its latches). Protection against trading on stale state is the composite stream-health posting gate + cancel-sweep, not a tick cascade. **Boot-time auto-approve** (`approvals.autoApprove: true`) reads the maker's `PositionModule` USDC allowance once before the first tick and raises it if short — but only after the boot-time state-loss hold has lifted (raising the allowance while the hold is active could re-activate latent soft-cancelled signed commitments; deferred until the hold clears, retried next tick). `mode: exact` (the safer default) raises to `min(risk-cap ceiling, current wallet USDC balance)` — wallet-bounded per DESIGN §6 so the allowance never overstates pullable risk; a `readBalances` failure fails closed. `mode: unlimited` (`MaxUint256`) skips the wallet-balance read and is gated by `--yes` on the CLI to keep it from being accidentally set. Raise-only: an existing allowance that already meets the target is left alone. Emits `approval` `{purpose, spender, currentAllowance, requiredAggregateAllowance, amountSetTo, walletBalanceWei6?, txHash, gasPolWei}` — `walletBalanceWei6` is present in exact mode and absent in unlimited. **Gas-budget verdict** (DESIGN §6): every on-chain write goes through `canSpendGas` first — refuses when today's accumulated POL spend (`state.dailyCounters[YYYY-MM-DD].gasPolWei`) plus `gas.emergencyReservePOL` would reach `gas.maxDailyGasPOL`, or on operator misconfig (reserve ≥ cap). Auto-approve is gated normally; settle/claim and the on-chain kill path may dip into the reserve via `mayUseReserve` when configured. A denied approve emits `candidate` `gas-budget-blocks-reapproval`; a denied settle/claim emits `candidate` `gas-budget-blocks-settlement` (with `purpose: 'settleSpeculation' | 'claimPosition'`). **Funding guard** (DESIGN §6): before posting each tick, re-reads `min(wallet USDC, PositionModule allowance)` (throttled to `fundingGuard.checkIntervalMs`) against the gross matchable-commitment risk; on a shortfall it **halts new posting** and, per `fundingGuard.underfundedCancelMode`, pulls existing visible quotes off the relay (`offchain` — `soft-cancel` `reason: 'funding'`) and/or authoritatively cancels them on chain (`onchain` — the only mode that clears the hold before expiry), emitting `funding-hold` on each enter/clear; a balance/allowance read failure fails closed into the hold (`failClosedOnReadError`). **Auto-settle + auto-claim** (`settlement.autoSettleOwn` / `autoClaimOwn`): each tick after the audit probes, the runner walks `state.positions` and (gas-gated, with `mayUseReserve = settlement.continueOnGasBudgetExhausted`) calls `ensureSpeculationSettled` (idempotent) for each `pendingSettle` record — our own settle tx emits `settle` with `winSide` / `txHash` / `gasPolWei`, while an already-settled or concurrent-race recovery emits a `candidate` `already-settled` skip instead (billing any reverted-race gas via `revertedReceipt`, never an `error`; only a genuine settle failure stays `error` `phase: 'settle'`) — and `ensurePositionClaimed` (idempotent) for each `claimable` record: our own claim tx emits `claim` with the event-sourced `payoutWei6` / `txHash` / `gasPolWei`, while an already-claimed or concurrent-race recovery emits a `candidate` `already-claimed` skip instead (no `claim` event and no payout — the contract zeroes economic fields post-claim, so none is derived; reverted-race gas billed via `revertedReceipt`; never an `error`, only a genuine `NotSettled` / `NoPayout` / RPC failure stays `error` `phase: 'claim'`). Every success outcome stamps the local status to `claimed` so the per-tick loop stops retrying. Realized P&L is wired into `summary` (unrealized — active positions marked to fair — is the remaining slice). Start tiny.
- **`ospex-mm summary [--since <ts>] [--json]`** — aggregates the NDJSON event logs under `telemetry.logDir` into the §2.3 run metrics: ticks, the candidate / event-kind histograms, quote-intent counts, the would-be-submit / replace / soft-cancel / expire tallies, the quote-competitiveness rate (and the vs-reference tick spread), the quote-age distribution (p50 / p90 / max), the latent-exposure peak, stale-quote incidents, and the error counts — plus the run window, run-ids, and (a `kill` event if any) shutdown reason. **Live-mode metrics** are populated when the log carries live events: **fill rate** (filled USDC ÷ quoted USDC across `submit` / `replace` / `fill`), **gas** (POL wei18 total + per-kind attribution across `approval` / `onchain-cancel` / `settle` / `claim`; optional USDC equivalent when `gas.reportInUSDC: true`), **settlements** (counts of `settle` / `claim` events + total claimed payout USDC), **realized P&L** (signed net + claimed-profit / realized-loss components, per-outcome counts — won / lost / push / won-unclaimed / already-claimed / unsettled — derived by correlating `claim.payoutWei6` against per-(speculationId, makerSide) cumulative stake from `fill` events plus `settle.winSide`; `already-claimed` (a position the auto-claim found already claimed out-of-window, via a `candidate already-claimed` skip rather than a `claim` event) is kept distinct from `won-unclaimed` and contributes no derived payout), and **estimated seed creation-fees** (USDC wei6 — a conservative estimate, not exact realized spend; `"0"` on any seeding-off / moneyline-only run). The fill rate and realized P&L are also **broken out per market** (`liveMetrics.fills.byMarket` / `liveMetrics.realizedPnl.byMarket` for `moneyline` / `spread` / `total`, each reconciling with the run-wide total); the text report adds a per-market line on a multi-market run. Unrealized P&L (active positions marked to current fair) is still ahead. `--since` windows it; `--json` emits a `{ schemaVersion: 1, summary: … }` envelope.
- `yarn mm --help` — the command list. (`yarn dev --help` does the same via `tsx`; `yarn build && yarn link`, then `ospex-mm --help`, puts the binary on your PATH.)
- The **pricing module** (`src/pricing/` — vig stripping, the economics/direct spread modes with their refusal paths, odds-tick conversion + bounds, sizing). Pure functions, unit-tested. Used by `quote`.
- The **config loader** (`src/config/` — `loadConfig` / `parseConfig`: parse the YAML, validate with strict unknown-key rejection, apply the `OSPEX_*` env overrides, default almost everything to `ospex-mm.example.yaml`'s values).
- The **risk engine** (`src/risk/` — worst-case-loss-by-outcome exposure accounting over positions + visible + latent commitments; the per-commitment / contest / team / sport / bankroll caps; the headroom and verdict functions; the `PositionModule` aggregate-allowance target). Pure functions, unit-tested. Used by `quote` and `doctor`.
- The **order-planning layer** (`src/orders/` — `buildDesiredQuote` turns config + reference odds + exposure headroom into a two-sided quote, which is what `quote --dry-run` runs; `inventoryFromState` translates the persisted state into the risk engine's `Inventory`; `reconcileBook` decides what to submit / replace / soft-cancel against the maker's current book on a speculation, per DESIGN §9). Pure functions, unit-tested. The runner wires these into the loop; the same path executes the writes in live mode (`run --live`) and simulates them in dry-run.
- The **runner** (`src/runners/` — the `Runner` class: the tick loop, the boot-time state-loss fail-safe (DESIGN §12), the kill-switch (a `KILL` file or a SIGTERM / SIGINT → a graceful shutdown that emits `kill` and flushes), **discovery** (every `discovery.everyNTicks` ticks — jittered — find the verified contests with an open speculation for a configured market + a reference-game id starting within `marketSelection.maxStartsWithinHours`, honour the allow/deny lists, track up to `marketSelection.maxTrackedMarkets` `(contest, market, line)` entries (earlier-listed `markets` win slots when it's tight), untrack the departed; `candidate` telemetry for the skipped/tracked), **reference-odds tracking** (DESIGN §10's stream guardrails — by default a core-api SSE odds stream per tracked market: snapshot-first seed, then `onChange` keeps the market's odds + freshness + a `dirty` flag current, `onRefresh` keeps freshness current, `onError` degrades the market; capped at `odds.maxRealtimeChannels` with the rest left degraded and retried when a slot frees; `odds.subscribe: false` falls back to a bounded per-tick snapshot poll; a degraded market is re-subscribed on the next discovery cycle; `degraded` telemetry), the **per-market reconcile** (DESIGN §3/§8/§9 — for each market that needs it [its reference odds moved, or it lacks a fresh two-sided standing quote of ours, throttled to roughly one re-quote per `orders.staleAfterSeconds`], and *not* while the boot-time state-loss hold is active: the unquoteable-market gates [`start-too-soon` / `stale-reference` (incl. a degraded odds channel) / `no-reference-odds`, plus `no-open-speculation` after a `getSpeculation` re-check] — each *pulls* (soft-cancels) any visible quote of ours on that speculation, since the visible book must never carry a quote the MM is no longer pricing (DESIGN §2.2 / §3), and logs the skip — otherwise `buildDesiredQuote` over the hypothetical inventory `inventoryFromState` derives from the persisted state, `reconcileBook` against the maker's book on that speculation, and — in dry-run — log it [`quote-intent` + `would-submit` / `would-replace` / `would-soft-cancel` + a `cap-hit` candidate per deferred side] and mutate the hypothetical inventory [synthetic `visibleOpen` records for would-be submits / replacements, pulled / replaced ones reclassified `softCancelled` — still counted against caps until expiry], then assess quote competitiveness [DESIGN §8 — for each would-be quote: where its tick would sit vs the visible orderbook on its side (the `getSpeculation` already fetched that book — no extra read) and vs the reference odds; a `quote-competitiveness` event per side, or one `competitiveness-unavailable` if that orderbook somehow isn't populated]; a `dirty` market is reconciled even when its quotes are fresh, and an unquoteable market with quotes still up is reconciled to pull them; a transient `getSpeculation` failure leaves the market to retry promptly next tick rather than wait out the stale-after throttle), age-out of expired tracked commitments → `expired` + an `expire` event, a prune of old terminal (`expired` / `filled` / `authoritativelyInvalidated`) commitment records past `max(3600, 10×orders.expirySeconds)` so a long shadow run's state file stays bounded, the per-tick state flush, and an interruptible sleep paced at `ownState.auditPollIntervalMs` (the runner's single tick cadence, default 60 s); the clock / sleep / kill-probe / signal / randomness seams — and the `OspexAdapter` itself (so `listContests` / `getContest` / `getOddsSnapshot` / `subscribeOdds` / `getSpeculation` can be faked) — are injectable, so it's unit-tested with bounded tick runs). The reconcile is **mode-aware**: under `--dry-run` it logs `would-*` events + mutates a hypothetical inventory; under `--live` (with the two-key match) it goes through `commitments.submitRaw` / `commitments.cancel` and records real commitment hashes. A failed live write logs `error` `phase:'submit'`/`'cancel'`, marks the market for prompt retry next tick, and never crashes the loop. In live mode the **own-state SSE stream** (owner-authenticated, always on — live boot refuses without a maker address) is the canonical writer of the maker's commitments, fills, and positions: fills arrive in real time, dirty the market, and emit `fill` / `position-transition` events. The per-tick **audit probes** (the `listOpenCommitments(maker)` diff, one aggregate `getPositionStatus(maker)`, per-hash probes of soft-cancelled records) converge a per-tick audit clone for the divergence comparator only; a probe failure marks the audit cycle failed (the comparator skips it, latches are preserved) and never gates the tick. A boot-time **auto-approve** raises the `PositionModule` USDC allowance to `min(risk-cap ceiling, current wallet USDC)` in `mode: exact` (or `MaxUint256` in `mode: unlimited`, gated by `--yes`), deferred while the state-loss hold is active; a **daily POL gas-budget verdict** (`canSpendGas` over `state.dailyCounters[YYYY-MM-DD].gasPolWei + gas.emergencyReservePOL` vs `gas.maxDailyGasPOL`) refuses on-chain writes once today's spend has eaten into the reserve (denied approves emit `candidate` `gas-budget-blocks-reapproval`; denied settle / claim emit `candidate` `gas-budget-blocks-settlement`). A **funding guard** (DESIGN §6) re-reads `min(wallet USDC, PositionModule allowance)` against the gross matchable-commitment risk before posting and **halts new quoting** on a shortfall — pulling existing quotes off the relay (`soft-cancel` `reason: 'funding'`) and/or authoritatively cancelling them on chain per `fundingGuard.underfundedCancelMode`, emitting `funding-hold` on each transition and failing closed into the hold on a read error. After the audit probes each tick, **auto-settle + auto-claim** (`settlement.autoSettleOwn` / `autoClaimOwn`) walk `state.positions` and dispatch idempotent `ensureSpeculationSettled` / `ensurePositionClaimed` for each `pendingSettle` / `claimable` record — gas-gated with `mayUseReserve = settlement.continueOnGasBudgetExhausted`. Own settle/claim txs emit `settle` / `claim` events carrying `winSide` / `payoutWei6` / `txHash` / `gasPolWei` and stamp `status: 'claimed'` on claim; an already-settled / already-claimed / concurrent-race recovery emits a `candidate` `already-settled` / `already-claimed` skip instead (no `claim` event or payout for already-claimed; reverted-race gas still billed; only genuine failures stay `error`). The **kill switch**: on an actual shutdown (KILL file or SIGTERM/SIGINT — not a `maxTicks` exit), an unconditional **off-chain sweep** pulls every visible quote (`visibleOpen` + `partiallyFilled`) via gasless `cancelCommitmentOffchain` — emits `soft-cancel` `reason: 'shutdown'`, reclassifies the records `softCancelled`. Then, if `killCancelOnChain: true`, the **on-chain authoritative sweep** runs: every non-terminal commitment (`visibleOpen` / `softCancelled` / `partiallyFilled`) gets `cancelCommitmentOnchain`'d — gas-gated with `mayUseReserve: true` (operator-explicit "burn the reserve"), denials emit `candidate` `gas-budget-blocks-onchain-cancel` and break the loop, successful cancels stamp the local record `authoritativelyInvalidated`. The `raiseMinNonce` bulk-invalidate optimization for `cancel-stale --authoritative` + the on-chain kill path, and unrealized P&L, are still ahead.
- The **`@ospex/sdk` adapter** (`src/ospex/` — the only module that imports `@ospex/sdk`; read-only wrappers over contests / games / speculations / commitments / positions / odds / balances / approvals / health; maps the SDK's provider-specific reference-game field to the neutral `referenceGameId` and drops the games surface's provider-named `externalIds` at this boundary). Used by `doctor`, `quote`, and `candidates`.
- The **state store** (`src/state/` — persisted-inventory shape, atomic JSON writes, the boot-time state-loss fail-safe; `doctor` reads it for the integrity check) and the **telemetry layer** (`src/telemetry/` — the NDJSON event-log writer + the `kind` vocabulary, wired into `run`; plus `summarize` / `listRunLogs`, behind `ospex-mm summary`).
- The design (`docs/DESIGN.md`), the annotated config (`ospex-mm.example.yaml`), and the safety checklist (`docs/OPERATOR_SAFETY.md`).

What's **not** implemented yet (see `docs/DESIGN.md §14`):

- **Unrealized P&L** — still ahead. Active positions marked to current fair value; requires `summary` to accept an `OspexAdapter` (a qualitative signature change). Realized P&L is wired. The `raiseMinNonce` bulk-invalidate is a future optimization for the on-chain cancel paths (currently all per-commitment).

So: the entire CLI surface (`doctor`, `quote --dry-run`, `candidates`, `run --dry-run`, `run --live`, `cancel-stale`, `status`, `summary`) is wired and full-featured. `run --live` *executes* (submits + off-chain cancels + stream-driven fill/position tracking + boot-time auto-approve + daily POL gas-budget verdict + auto-settle + auto-claim + on-chain kill on shutdown), and `summary` populates live-mode metrics including realized P&L; unrealized P&L from API marks is the remaining enhancement. The live path has been exercised end-to-end on Polygon mainnet in small, controlled runs — but treat it as alpha: dry-run first, tiny caps, and the [`docs/OPERATOR_SAFETY.md`](docs/OPERATOR_SAFETY.md) checklist before going live.

## What this is

- A clone-configure-run **market maker template** for Ospex. The minimum that lets an agent safely **quote, update, cancel, get filled, settle, claim, and produce metrics** — with strict bankroll controls and a clear safety model.
- Built **on `@ospex/sdk`** — the SDK provides the contract/API plumbing (EIP-712 signing, reads, submit/match/cancel, positions, the odds SSE subscription, typed errors, a nonce counter); this repo owns the *decisions* (which markets to quote, at what price and size, when to cancel/replace, when to stop) plus the config, risk engine, event loop, persistent state, and telemetry.
- **Subscription-first** for reference odds (it streams price changes rather than polling per contest) so it doesn't hammer the public API — and for the maker's *own* state: in live mode the owner-authenticated own-state SSE stream delivers fills, commitment lifecycle, and position status in real time; a bounded per-tick poll of the maker's own commitments/positions survives only as an audit cross-check.

## What this is *not*

- Not a sophisticated quant system — it does not try to beat sportsbooks. The pricing is a first-order model with honestly-flagged assumptions (see `docs/DESIGN.md §5`).
- Not a speculation seeder — v0 quotes only speculations that already exist; it does not create new ones.
- Not the home for flow / taker agents, observer agents, or cross-agent scorecards — those exist to *exercise* a market maker and *prove the platform*; they live in a separate test harness, not here.

## Wallet model

Ospex never asks for your private key. Use a [Foundry](https://book.getfoundry.sh/) keystore:

```bash
mkdir -p ~/.foundry/keystores
cast wallet new ~/.foundry/keystores ospex-mm    # Foundry generates the key, prints only the address
                                                  # — or `cast wallet import ospex-mm` for an existing key
```

Point the config at that file with an **absolute path** (`wallet.keystorePath`, or the `OSPEX_KEYSTORE_PATH` env var) — the path is used verbatim, so `~` is not expanded. A keystore is only needed for live mode: `doctor --address <0x…>`, `quote --dry-run`, and `run --dry-run` sign nothing, so `wallet.keystorePath` can stay blank until you go live. You also bring your own RPC URL — Alchemy, Infura, or QuickNode (the public Polygon RPCs are rate-limited and unreliable; there is no default). The MM prompts for the keystore passphrase only when it needs to sign.

## Quick start

```bash
yarn install
yarn build

cp ospex-mm.example.yaml ospex-mm.yaml
# edit ospex-mm.yaml — rpcUrl, pricing.economics (capital + target return), risk caps.
# wallet.keystorePath can stay blank until you go live — nothing before `run --live` signs anything.

yarn mm doctor --address <0xYou>     # readiness: balances (USDC + POL), PositionModule allowance, API/RPC, state — fully read-only
                                     #   (--address is optional — without it the chain checks SKIP; if you pass one, use the
                                     #    wallet you'll actually fund — doctor FAILs a 0-POL wallet)
yarn mm candidates                   # read-only preflight: what's quotable right now + which upcoming games can
                                     #   become quotable contests (signer-free; --json for a machine-readable envelope)
yarn mm quote --dry-run <contestId>  # one-shot: fetch reference odds, compute a two-sided quote, print the breakdown
                                     #   (pick a contestId from `yarn mm candidates` — see "Find a quote target" in docs/QUICKSTART.md §5)
yarn mm run --dry-run                # shadow mode: the full loop, posts nothing — let it run a while
yarn mm summary                      # aggregate the NDJSON event logs into the run metrics — read this before going live
# then, deliberately: set wallet.keystorePath (absolute path) and mode.dryRun: false in the config
# AND pass --live (the two-key model); OSPEX_KEYSTORE_PASSPHRASE=… in the env, else a no-echo TTY prompt
OSPEX_KEYSTORE_PASSPHRASE=… yarn mm run --live
```

`yarn mm <cmd>` runs the built CLI (`node dist/cli/index.js`). `yarn dev <cmd>` runs it via `tsx` without a build, for iteration. To get the `ospex-mm` binary on your PATH: `yarn build && yarn link` (or `npm link`), then `ospex-mm <cmd>`. The full command surface — `doctor`, `quote --dry-run`, `candidates`, `run --dry-run`, `run --live` (alpha; see *Current status*), `cancel-stale [--authoritative]`, `status`, and `summary` — is wired.

## Configuration

The annotated reference config is **[`ospex-mm.example.yaml`](ospex-mm.example.yaml)** — copy it, fill in rpc / pricing / risk (wallet only when you go live), run. A novice typically touches only `rpcUrl`, `pricing.economics`, and maybe `risk` — plus `wallet` when going live. The full schema and the rationale for every field are in `docs/DESIGN.md §7`. Defaults are conservative and `dryRun: true`.

## Architecture (brief — full design in `docs/DESIGN.md`)

A single long-running **worker** process. No HTTP surface. Runs locally (`ospex-mm run`) or as a worker dyno. Layers under `src/`: `config/` (load + validate), `ospex/` (the only module that imports `@ospex/sdk`), `pricing/` (vig strip → fair value → spread → quote prices), `risk/` (worst-case-loss-by-outcome accounting + cap enforcement + the allowance target), `orders/` (the order lifecycle), `state/` (persistent inventory, JSON, single-process), `telemetry/` (NDJSON event log + the `summary` aggregator), `runners/` (the event loop), `cli/` (`doctor | quote | run | cancel-stale | status | summary`).

## Safety — read this

This is **experimental software**, provided **without warranty**. Wagering and on-chain transactions carry **financial risk**, and there is **no profit guarantee** — odds accuracy, liquidity, settlement timing, RPC availability, and indexer latency are all best-effort. Nothing here is financial, legal, or tax advice; **you are responsible for compliance with the laws of your jurisdiction** and any tax/regulatory treatment — this software enforces no geofencing or KYC. **Dry-run first; start with tiny caps.** The MM never logs your private key, but when live it *will* submit transactions and move funds; approvals can be abused by buggy software — approve only what you'll risk, and audit with `ospex doctor` / `ospex approvals show`. An off-chain cancel pulls a quote from the order book but does **not** invalidate the signed payload — a taker holding it can still match it until it expires (≈2 minutes by default); for a hard stop use the on-chain kill (`killCancelOnChain: true`, costs gas). The full checklist is in **[`docs/OPERATOR_SAFETY.md`](docs/OPERATOR_SAFETY.md)**.

## Development

```bash
yarn smoke          # runs install / build / typecheck / lint / test / mm --help in series; fails fast on first error
                    # — or run the steps individually:
yarn install        # pulls @ospex/sdk from its pinned GitHub Release tarball + transitive deps
                    # — package.json / yarn.lock are the source of truth for the pinned version (v0.6.2 at the time of writing)
yarn build          # tsc -> dist/
yarn typecheck      # tsc --noEmit
yarn test           # vitest
yarn lint           # eslint src
```

Branches and PRs only — never commit to `main`. See **[`CONTRIBUTING.md`](CONTRIBUTING.md)**.
