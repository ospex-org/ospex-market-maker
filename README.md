# ospex-market-maker

A **reference market maker** for [Ospex](https://ospex.org) ŌĆö a zero-vig peer-to-peer sports prediction protocol on Polygon. Clone it, point it at your wallet, bankroll, RPC URL, and return target, and it quotes two-sided liquidity on upcoming contests, manages its exposure, reacts to fills, settles and claims, and writes an auditable record of everything it did.

It is built on **[`@ospex/sdk`](https://github.com/ospex-org/ospex-sdk)** ŌĆö every chain and API interaction goes through the SDK; this repo never calls the Ospex contracts directly.

- **`ospex-mm run --dry-run`** ŌĆö the shadow loop: boots through the state-loss fail-safe, then runs the tick loop ŌĆö discovery ŌåÆ reference-odds tracking ŌåÆ the per-market reconcile ŌåÆ age-out ŌåÆ state flush ŌĆö until a kill-switch file appears or a SIGTERM / SIGINT arrives; logs `quote-intent` / `quote-competitiveness` / `would-submit` / `would-replace` / `would-soft-cancel` / `candidate` / `expire` / `degraded` / `error` / `kill` events to the NDJSON log under `telemetry.logDir`, and mutates a *hypothetical* inventory ŌĆö but **posts nothing on chain**. `--address` / `--keystore` (overrides the configured path) / `--ignore-missing-state` (attests no prior run left a still-matchable commitment ŌĆö lifts the boot-time hold).
- **`ospex-mm cancel-stale [--authoritative]`** ŌĆö one-shot operator cleanup: pulls every tracked commitment whose age exceeds `orders.staleAfterSeconds`. By default it runs the gasless off-chain DELETE for each stale `visibleOpen` (records ŌåÆ `softCancelled`, emits `soft-cancel` `reason: 'stale'`); a `partiallyFilled` remainder is **skipped** off-chain ŌĆö the API rejects a DELETE once a commitment has matched (`409 COMMITMENT_MATCHED`), so it's counted in `offchainSkippedPartial` and left in place (its `filledRiskWei6` untouched) ŌĆö and `softCancelled` records are a no-op off-chain, so for either only `--authoritative` invalidates the still-matchable signed payload. With `--authoritative` it *also* calls `cancelCommitmentOnchain` per non-terminal record ŌĆö including `partiallyFilled` (records ŌåÆ `authoritativelyInvalidated`, emits `onchain-cancel`) ŌĆö gas-gated with `mayUseReserve: true` (operator-explicit "burn the reserve to kill latent exposure"), denials emit `candidate` `gas-budget-blocks-onchain-cancel` and break the loop. Stop a running `run --live` first ŌĆö the JSON state file is not multi-process safe. Same two-key principle as `run --live` (requires `mode.dryRun: false` in the config; the keystore passphrase comes from `OSPEX_KEYSTORE_PASSPHRASE` or a TTY prompt). Mirrors the runner's boot-time state-loss fail-safe (DESIGN ┬¦12): refuses if the state file is missing but prior telemetry exists, unless `--ignore-missing-state` is passed, and refuses on a dry-run synthetic hash (`dry:ŌĆ”`) in the loaded state. Both checks run *before* the keystore unlock so a doomed invocation never prompts for a passphrase. `--json` emits a `{ schemaVersion: 1, cancelStale: ŌĆ” }` envelope; exits non-zero if any record failed to cancel (`errored > 0`) or a gas-budget denial blocked the on-chain leg (`gasDenied > 0`).
- **`ospex-mm status [--address <0xŌĆ”>] [--json]`** ŌĆö read-only snapshot of the persisted state: commitments by lifecycle (`visibleOpen` / `softCancelled` / `partiallyFilled` / `filled` / `expired` / `authoritativelyInvalidated` counts + distinct contests with non-terminal exposure), positions by local `MakerPositionStatus` (counts + per-bucket own-risk USDC sums), today + lifetime gas spent (POL) / fees paid (USDC), the PnL snapshot, last flushed-at + last run id, and ŌĆö when a maker address resolves without decrypting the keystore (`--address`, or a v3 keystore with a plaintext `address` field) ŌĆö the SDK's `positions.status(maker)` totals from the API (signer-free read). No keystore unlock; no writes; always exits 0. State-loss situations (missing-but-prior-telemetry, corrupt file) are **reported**, not refused ŌĆö the integrity field surfaces the diagnostic so the operator can see *why* their MM is misbehaving. `--json` emits a `{ schemaVersion: 1, status: ŌĆ” }` envelope.
- **`ospex-mm run --live`** ŌĆö the same loop, with the reconcile's submits going through `commitments.submitRaw` and the off-chain cancels through `commitments.cancel` (the protocol tuple comes from `toProtocolQuote`; nonce auto-picked; the short fixed-seconds / match-time `expiry` is passed explicitly). Gated by the two-key model (DESIGN ┬¦8): requires *both* `mode.dryRun: false` in the config *and* the `--live` flag; otherwise refused with a clear message. Reads the keystore passphrase from `OSPEX_KEYSTORE_PASSPHRASE` (preferred for non-interactive runs), else prompts no-echo on a TTY. The signer's address determines the maker wallet ŌĆö `--address` with `--live` is refused. The Runner ctor also refuses to boot live on a state directory containing dry-run synthetic commitments (use a fresh `state.dir` for live). Events log as `submit` / `replace` / `soft-cancel` (instead of the `would-*` counterparts), carrying the real commitment hashes. **Fill detection** (DESIGN ┬¦10) runs each tick before the reconcile: `listOpenCommitments(maker)` is diffed against the local visibleOpen / partiallyFilled set (including past-local-expiry records, so a fill landing right before expiry isn't missed by `ageOut`'s clock-only terminalization), disappeared hashes are classified via `getCommitment(hash)` with any unobserved fill delta applied *first* (a commitment can have partially filled and then expired / been cancelled between polls), and partial-fill bumps are detected on still-listed commitments; the affected market is dirtied so the same tick re-prices the imbalance. **Fails closed**: if `listOpenCommitments` throws, the tick skips reconcile, the position poll, AND `ageOut` ŌĆö the runner doesn't write or terminalize on unverified fill state, and the markets stay dirty for next-tick retry. A separate **position-status poll** (one `getPositionStatus(maker)` per tick, after fill detection) backfills `MakerPositionRecord`s the commitment-list diff can't see ŌĆö a taker matching a soft-cancelled commitment via a stale signed payload before expiry shows up on the position side and gets recorded then ŌĆö AND mirrors each polled position's API bucket (`active` / `pendingSettle` / `claimable`) into the local `MakerPositionStatus`, emitting a `position-transition` event on each forward step (`claimed` is stamped later by the auto-claim path). **Boot-time auto-approve** (`approvals.autoApprove: true`) reads the maker's `PositionModule` USDC allowance once before the first tick and raises it if short ŌĆö but only after the boot-time state-loss hold has lifted (raising the allowance while the hold is active could re-activate latent soft-cancelled signed commitments; deferred until the hold clears, retried next tick). `mode: exact` (the safer default) raises to `min(risk-cap ceiling, current wallet USDC balance)` ŌĆö wallet-bounded per DESIGN ┬¦6 so the allowance never overstates pullable risk; a `readBalances` failure fails closed. `mode: unlimited` (`MaxUint256`) skips the wallet-balance read and is gated by `--yes` on the CLI to keep it from being accidentally set. Raise-only: an existing allowance that already meets the target is left alone. Emits `approval` `{purpose, spender, currentAllowance, requiredAggregateAllowance, amountSetTo, walletBalanceWei6?, txHash, gasPolWei}` ŌĆö `walletBalanceWei6` is present in exact mode and absent in unlimited. **Gas-budget verdict** (DESIGN ┬¦6): every on-chain write goes through `canSpendGas` first ŌĆö refuses when today's accumulated POL spend (`state.dailyCounters[YYYY-MM-DD].gasPolWei`) plus `gas.emergencyReservePOL` would reach `gas.maxDailyGasPOL`, or on operator misconfig (reserve Ōēź cap). Auto-approve is gated normally; settle/claim and the on-chain kill path may dip into the reserve via `mayUseReserve` when configured. Every success outcome stamps the local status to `claimed` so the per-tick loop stops retrying. P&L is still landing in a later Phase-3 slice ŌĆö start tiny, on testnet only, until it lands.

## What this is

- A clone-configure-run **market maker template** for Ospex. The minimum that lets an agent safely **quote, update, cancel, get filled, settle, claim, and produce metrics** ŌĆö with strict bankroll controls and a clear safety model.
- Built **on `@ospex/sdk`** ŌĆö the SDK provides the contract/API plumbing (EIP-712 signing, reads, submit/match/cancel, positions, the odds SSE subscription, typed errors, a nonce counter); this repo owns the *decisions* (which markets to quote, at what price and size, when to cancel/replace, when to stop) plus the config, risk engine, event loop, persistent state, and telemetry.
- **Subscription-first** for reference odds (it streams price changes rather than polling per contest) so it doesn't hammer the public API. Fill detection is bounded polling of the maker's *own* commitments/positions for now; an SSE fills stream is a planned follow-up.

## What this is *not*

- Not a sophisticated quant system ŌĆö it does not try to beat sportsbooks. The pricing is a first-order model with honestly-flagged assumptions (see `docs/DESIGN.md ┬¦5`).
- Not a speculation seeder ŌĆö v0 quotes only speculations that already exist; it does not create new ones.
- Not the home for flow / taker agents, observer agents, or cross-agent scorecards ŌĆö those exist to *exercise* a market maker and *prove the platform*; they live in a separate test harness, not here.

## Wallet model

Ospex never asks for your private key. Use a [Foundry](https://book.getfoundry.sh/) keystore:

```bash
mkdir -p ~/.foundry/keystores
cast wallet new ~/.foundry/keystores ospex-mm    # Foundry generates the key, prints only the address
                                                  # ŌĆö or `cast wallet import ospex-mm` for an existing key
```

Point the config at that path (`wallet.keystorePath`, or the `OSPEX_KEYSTORE_PATH` env var). You also bring your own RPC URL ŌĆö Alchemy, Infura, or QuickNode (the public Polygon RPCs are rate-limited and unreliable; there is no default). The MM prompts for the keystore passphrase only when it needs to sign.

## Architecture (brief ŌĆö full design in `docs/DESIGN.md`)

A single long-running **worker** process. No HTTP surface. Runs locally (`ospex-mm run`) or as a worker dyno. Layers under `src/`: `config/` (load + validate), `ospex/` (the only module that imports `@ospex/sdk`), `pricing/` (vig strip ŌåÆ fair value ŌåÆ spread ŌåÆ quote prices), `risk/` (worst-case-loss-by-outcome accounting + cap enforcement + the allowance target), `orders/` (the order lifecycle), `state/` (persistent inventory, JSON, single-process), `telemetry/` (NDJSON event log + the `summary` aggregator), `runners/` (the event loop), `cli/` (`doctor | quote | run | cancel-stale | status | summary`).
