# ospex-market-maker

A **reference market maker** for [Ospex](https://ospex.org) тАФ a zero-vig peer-to-peer sports prediction protocol on Polygon. Clone it, point it at your wallet, bankroll, RPC URL, and return target, and it quotes two-sided liquidity on upcoming contests, manages its exposure, reacts to fills, settles and claims, and writes an auditable record of everything it did.

It is built on **[`@ospex/sdk`](https://github.com/ospex-org/ospex-sdk)** тАФ every chain and API interaction goes through the SDK; this repo never calls the Ospex contracts directly.

It remains **experimental software**: run it in dry-run first, go live deliberately with tiny caps, and read **[`docs/OPERATOR_SAFETY.md`](docs/OPERATOR_SAFETY.md)** before risking funds. ## Current status

What works:

- `yarn install && yarn build && yarn typecheck && yarn lint && yarn test` тАФ clean (793 unit tests across the pricing, config, risk, orders, state, telemetry, ospex-adapter, reducer, runner, and CLI-command modules).
- **`ospex-mm doctor`** тАФ readiness probe: config, keystore, API, RPC, POL/USDC balances, `PositionModule` allowance, and the persisted-state integrity check, plus a "Ready to" matrix (dry-run shadow тАФ needs no keystore, since the shadow loop posts nothing / post commitments тАФ the live prereqs). `--address <0xтАж>` keeps it fully read-only (no passphrase prompt); `--json` emits a `{ schemaVersion: 1, doctor: тАж }` envelope.
- **`ospex-mm quote --dry-run <contestId>`** тАФ computes a two-sided moneyline quote breakdown (reference odds тЖТ fair value тЖТ spread тЖТ priced quote with size), or refuses with a clear message (contest closed, no open moneyline speculation, no reference odds). Never posts; `--json` emits a `{ schemaVersion: 1, quote: тАж }` envelope.
- **`ospex-mm run --dry-run`** тАФ the shadow loop: boots through the state-loss fail-safe, then runs the tick loop тАФ discovery тЖТ reference-odds tracking тЖТ the per-market reconcile тЖТ age-out тЖТ state flush тАФ until a kill-switch file appears or a SIGTERM / SIGINT arrives; logs `quote-intent` / `quote-competitiveness` / `would-submit` / `would-replace` / `would-soft-cancel` / `candidate` / `expire` / `degraded` / `error` / `kill` events to the NDJSON log under `telemetry.logDir`, and mutates a *hypothetical* inventory тАФ but **posts nothing on chain**. `--address` / `--keystore` (overrides the configured path) / `--ignore-missing-state` (attests no prior run left a still-matchable commitment тАФ lifts the boot-time hold).
- **`ospex-mm cancel-stale [--authoritative]`** тАФ one-shot operator cleanup: pulls every tracked commitment whose age exceeds `orders.staleAfterSeconds`. By default it runs the gasless off-chain DELETE for each stale `visibleOpen` (records тЖТ `softCancelled`, emits `soft-cancel` `reason: 'stale'`); a `partiallyFilled` remainder is **skipped** off-chain тАФ the API rejects a DELETE once a commitment has matched (`409 COMMITMENT_MATCHED`), so it's counted in `offchainSkippedPartial` and left in place (its `filledRiskWei6` untouched) тАФ and `softCancelled` records are a no-op off-chain, so for either only `--authoritative` invalidates the still-matchable signed payload. With `--authoritative` it *also* calls `cancelCommitmentOnchain` per non-terminal record тАФ including `partiallyFilled` (records тЖТ `authoritativelyInvalidated`, emits `onchain-cancel`) тАФ gas-gated with `mayUseReserve: true` (operator-explicit "burn the reserve to kill latent exposure"), denials emit `candidate` `gas-budget-blocks-onchain-cancel` and break the loop. Stop a running `run --live` first тАФ the JSON state file is not multi-process safe. Same two-key principle as `run --live` (requires `mode.dryRun: false` in the config; the keystore passphrase comes from `OSPEX_KEYSTORE_PASSPHRASE` or a TTY prompt). Mirrors the runner's boot-time state-loss fail-safe (DESIGN ┬з12): refuses if the state file is missing but prior telemetry exists, unless `--ignore-missing-state` is passed, and refuses on a dry-run synthetic hash (`dry:тАж`) in the loaded state. Both checks run *before* the keystore unlock so a doomed invocation never prompts for a passphrase. `--json` emits a `{ schemaVersion: 1, cancelStale: тАж }` envelope; exits non-zero if any record failed to cancel (`errored > 0`) or a gas-budget denial blocked the on-chain leg (`gasDenied > 0`).
- **`ospex-mm status [--address <0xтАж>] [--json]`** тАФ read-only snapshot of the persisted state: commitments by lifecycle (`visibleOpen` / `softCancelled` / `partiallyFilled` / `filled` / `expired` / `authoritativelyInvalidated` counts + distinct contests with non-terminal exposure), positions by local `MakerPositionStatus` (counts + per-bucket own-risk USDC sums), today + lifetime gas spent (POL) / fees paid (USDC), the PnL snapshot, last flushed-at + last run id, and тАФ when a maker address resolves without decrypting the keystore (`--address`, or a v3 keystore with a plaintext `address` field) тАФ the SDK's `positions.status(maker)` totals from the API (signer-free read). No keystore unlock; no writes; always exits 0. State-loss situations (missing-but-prior-telemetry, corrupt file) are **reported**, not refused тАФ the integrity field surfaces the diagnostic so the operator can see *why* their MM is misbehaving. `--json` emits a `{ schemaVersion: 1, status: тАж }` envelope.
- **`ospex-mm run --live`** тАФ the same loop, with the reconcile's submits going through `commitments.submitRaw` and the off-chain cancels through `commitments.cancel` (the protocol tuple comes from `toProtocolQuote`; nonce auto-picked; the short fixed-seconds / match-time `expiry` is passed explicitly). Gated by the two-key model (DESIGN ┬з8): requires *both* `mode.dryRun: false` in the config *and* the `--live` flag; otherwise refused with a clear message. Reads the keystore passphrase from `OSPEX_KEYSTORE_PASSPHRASE` (preferred for non-interactive runs), else prompts no-echo on a TTY. The signer's address determines the maker wallet тАФ `--address` with `--live` is refused. The Runner ctor also refuses to boot live on a state directory containing dry-run synthetic commitments (use a fresh `state.dir` for live). Events log as `submit` / `replace` / `soft-cancel` (instead of the `would-*` counterparts), carrying the real commitment hashes. **Own-state is stream-driven** (DESIGN ┬з10): the owner-authenticated own-state SSE stream тАФ always on in live mode тАФ is the canonical writer of the maker's commitments, fills, and positions, so fills and lifecycle changes land in real time (the affected market is dirtied and re-priced by the next reconcile), `fill` events carry `source: 'own-state-stream'`, and each position's forward status step emits a `position-transition` event (`claimed` is stamped later by the auto-claim path). The per-tick **audit probes** тАФ the `listOpenCommitments(maker)` diff, one aggregate `getPositionStatus(maker)`, and per-hash probes of soft-cancelled records тАФ survive only as a slow cross-check: they converge a per-tick audit clone for the divergence comparator and never write canonical state or gate the tick (a probe failure marks the audit cycle failed; the comparator skips it and preserves its latches). Protection against trading on stale state is the composite stream-health posting gate + cancel-sweep, not a tick cascade. **Boot-time auto-approve** (`approvals.autoApprove: true`) reads the maker's `PositionModule` USDC allowance once before the first tick and raises it if short тАФ but only after the boot-time state-loss hold has lifted (raising the allowance while the hold is active could re-activate latent soft-cancelled signed commitments; deferred until the hold clears, retried next tick). `mode: exact` (the safer default) raises to `min(risk-cap ceiling, current wallet USDC balance)` тАФ wallet-bounded per DESIGN ┬з6 so the allowance never overstates pullable risk; a `readBalances` failure fails closed. `mode: unlimited` (`MaxUint256`) skips the wallet-balance read and is gated by `--yes` on the CLI to keep it from being accidentally set. Raise-only: an existing allowance that already meets the target is left alone. Emits `approval` `{purpose, spender, currentAllowance, requiredAggregateAllowance, amountSetTo, walletBalanceWei6?, txHash, gasPolWei}` тАФ `walletBalanceWei6` is present in exact mode and absent in unlimited. **Gas-budget verdict** (DESIGN ┬з6): every on-chain write goes through `canSpendGas` first тАФ refuses when today's accumulated POL spend (`state.dailyCounters[YYYY-MM-DD].gasPolWei`) plus `gas.emergencyReservePOL` would reach `gas.maxDailyGasPOL`, or on operator misconfig (reserve тЙе cap). Auto-approve is gated normally; settle/claim and the on-chain kill path may dip into the reserve via `mayUseReserve` when configured. A denied approve emits `candidate` `gas-budget-blocks-reapproval`; a denied settle/claim emits `candidate` `gas-budget-blocks-settlement` (with `purpose: 'settleSpeculation' | 'claimPosition'`). **Funding guard** (DESIGN ┬з6): before posting each tick, re-reads `min(wallet USDC, PositionModule allowance)` (throttled to `fundingGuard.checkIntervalMs`) against the gross matchable-commitment risk; on a shortfall it **halts new posting** and, per `fundingGuard.underfundedCancelMode`, pulls existing visible quotes off the relay (`offchain` тАФ `soft-cancel` `reason: 'funding'`) and/or authoritatively cancels them on chain (`onchain` тАФ the only mode that clears the hold before expiry), emitting `funding-hold` on each enter/clear; a balance/allowance read failure fails closed into the hold (`failClosedOnReadError`). **Auto-settle + auto-claim** (`settlement.autoSettleOwn` / `autoClaimOwn`): each tick after the audit probes, the runner walks `state.positions` and (gas-gated, with `mayUseReserve = settlement.continueOnGasBudgetExhausted`) calls `ensureSpeculationSettled` (idempotent) for each `pendingSettle` record тАФ our own settle tx emits `settle` with `winSide` / `txHash` / `gasPolWei`, while an already-settled or concurrent-race recovery emits a `candidate` `already-settled` skip instead (billing any reverted-race gas via `revertedReceipt`, never an `error`; only a genuine settle failure stays `error` `phase: 'settle'`) тАФ and `ensurePositionClaimed` (idempotent) for each `claimable` record: our own claim tx emits `claim` with the event-sourced `payoutWei6` / `txHash` / `gasPolWei`, while an already-claimed or concurrent-race recovery emits a `candidate` `already-claimed` skip instead (no `claim` event and no payout тАФ the contract zeroes economic fields post-claim, so none is derived; reverted-race gas billed via `revertedReceipt`; never an `error`, only a genuine `NotSettled` / `NoPayout` / RPC failure stays `error` `phase: 'claim'`). Every success outcome stamps the local status to `claimed` so the per-tick loop stops retrying. Realized P&L is wired into `summary` (unrealized тАФ active positions marked to fair тАФ is the remaining slice). Start tiny.
- **`ospex-mm summary [--since <ts>] [--json]`** тАФ aggregates the NDJSON event logs under `telemetry.logDir` into the ┬з2.3 run metrics: ticks, the candidate / event-kind histograms, quote-intent counts, the would-be-submit / replace / soft-cancel / expire tallies, the quote-competitiveness rate (and the vs-reference tick spread), the quote-age distribution (p50 / p90 / max), the latent-exposure peak, stale-quote incidents, and the error counts тАФ plus the run window, run-ids, and (a `kill` event if any) shutdown reason. **Live-mode metrics** are populated when the log carries live events: **fill rate** (filled USDC ├╖ quoted USDC across `submit` / `replace` / `fill`), **gas** (POL wei18 total + per-kind attribution across `approval` / `onchain-cancel` / `settle` / `claim`; optional USDC equivalent when `gas.reportInUSDC: true`), **settlements** (counts of `settle` / `claim` events + total claimed payout USDC), **realized P&L** (signed net + claimed-profit / realized-loss components, per-outcome counts тАФ won / lost / push / won-unclaimed / already-claimed / unsettled тАФ derived by correlating `claim.payoutWei6` against per-(speculationId, makerSide) cumulative stake from `fill` events plus `settle.winSide`; `already-claimed` (a position the auto-claim found already claimed out-of-window, via a `candidate already-claimed` skip rather than a `claim` event) is kept distinct from `won-unclaimed` and contributes no derived payout), and **fees** (USDC wei6 тАФ genuinely `"0"` in v0). Unrealized P&L (active positions marked to current fair) is still ahead. `--since` windows it; `--json` emits a `{ schemaVersion: 1, summary: тАж }` envelope.

## What this is

- A clone-configure-run **market maker template** for Ospex. The minimum that lets an agent safely **quote, update, cancel, get filled, settle, claim, and produce metrics** тАФ with strict bankroll controls and a clear safety model.
- Built **on `@ospex/sdk`** тАФ the SDK provides the contract/API plumbing (EIP-712 signing, reads, submit/match/cancel, positions, the odds SSE subscription, typed errors, a nonce counter); this repo owns the *decisions* (which markets to quote, at what price and size, when to cancel/replace, when to stop) plus the config, risk engine, event loop, persistent state, and telemetry.
- **Subscription-first** for reference odds (it streams price changes rather than polling per contest) so it doesn't hammer the public API тАФ and for the maker's *own* state: in live mode the owner-authenticated own-state SSE stream delivers fills, commitment lifecycle, and position status in real time; a bounded per-tick poll of the maker's own commitments/positions survives only as an audit cross-check.

## What this is *not*

- Not a sophisticated quant system тАФ it does not try to beat sportsbooks. The pricing is a first-order model with honestly-flagged assumptions (see `docs/DESIGN.md ┬з5`).
- Not a speculation seeder тАФ v0 quotes only speculations that already exist; it does not create new ones.
- Not the home for flow / taker agents, observer agents, or cross-agent scorecards тАФ those exist to *exercise* a market maker and *prove the platform*; they live in a separate test harness, not here.

## Wallet model

Ospex never asks for your private key. Use a [Foundry](https://book.getfoundry.sh/) keystore:

```bash
mkdir -p ~/.foundry/keystores
cast wallet new ~/.foundry/keystores ospex-mm    # Foundry generates the key, prints only the address
                                                  # тАФ or `cast wallet import ospex-mm` for an existing key
```

Point the config at that path (`wallet.keystorePath`, or the `OSPEX_KEYSTORE_PATH` env var). You also bring your own RPC URL тАФ Alchemy, Infura, or QuickNode (the public Polygon RPCs are rate-limited and unreliable; there is no default). The MM prompts for the keystore passphrase only when it needs to sign.

## Architecture (brief тАФ full design in `docs/DESIGN.md`)

A single long-running **worker** process. No HTTP surface. Runs locally (`ospex-mm run`) or as a worker dyno. Layers under `src/`: `config/` (load + validate), `ospex/` (the only module that imports `@ospex/sdk`), `pricing/` (vig strip тЖТ fair value тЖТ spread тЖТ quote prices), `risk/` (worst-case-loss-by-outcome accounting + cap enforcement + the allowance target), `orders/` (the order lifecycle), `state/` (persistent inventory, JSON, single-process), `telemetry/` (NDJSON event log + the `summary` aggregator), `runners/` (the event loop), `cli/` (`doctor | quote | run | cancel-stale | status | summary`).
